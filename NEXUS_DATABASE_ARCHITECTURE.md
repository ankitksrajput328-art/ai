# 🗄️ Nexus AI: High-Performance Database Architecture

This document defines the relational and vector data structures required to power a global AI ecosystem.

---

## 1. Dual-Database Strategy
- **PostgreSQL:** For structured relational data (Users, Subscriptions, Chat Metadata).
- **Pinecone (Vector):** For unstructured semantic data (Conversation embeddings, Document fragments).

---

## 2. PostgreSQL Schema Overview

### `users` table
- `id`: UUID (Primary Key)
- `email`: String (Unique)
- `password_hash`: String
- `is_premium`: Boolean
- `credits`: Integer
- `created_at`: Timestamp

### `chats` table
- `id`: UUID
- `user_id`: UUID (Foreign Key)
- `title`: String
- `summary`: Text (Generated by AI)
- `created_at`: Timestamp

### `memories` table (Metadata Layer)
- `id`: UUID
- `user_id`: UUID
- `chat_id`: UUID
- `content`: Text (Short summary of the fact)
- `vector_id`: String (Pointer to Pinecone ID)
- `importance`: Float (0.0 to 1.0)
- `recency_score`: Float

---

## 3. Vector Database (Pinecone) Design
- **Namespace:** `user_{id}` (Ensures strict data isolation).
- **Metadata Fields:**
    - `text`: The original conversation chunk.
    - `category`: (e.g., 'coding', 'personal', 'work').
    - `timestamp`: To allow recency-based filtering.

---

## 4. Optimization: The "Memory Compaction" Pipeline
1.  **Daily Job:** Scan for redundant memories (e.g., user said "I love Python" in 3 different chats).
2.  **Consolidation:** Use an LLM to merge these into a single high-importance memory: "User is a Python enthusiast."
3.  **Deletion:** Remove the original 3 low-score vectors to save space and speed up search.

---

**Architect's Note:** By keeping the "Facts" (Memories table) in PostgreSQL and the "Raw Semantics" in Pinecone, we get the best of both worlds—fast filtering and deep intelligence.
